# 需求描述补全优化方法论

本文档说明如何将用户原始需求描述补全为结构化、专业化的需求文档。

---

## 一、为什么要补全需求描述

用户提交的原始需求通常是碎片化、口语化、不完整的：
- "帮我做个自动化采集脚本"（缺价值量化、缺目标用户、缺成功标准）
- "界面按钮改一下"（缺背景、缺影响范围）
- "加个报表功能"（缺使用场景、缺预期价值）

不完整的描述会导致评分失真。补全环节确保AI基于**完备的需求信息**进行专业分析。

---

## 二、补全方法论

### 1. 5W1H 六要素分析

| 要素 | 问题 | 对应字段 | 示例 |
|------|------|----------|------|
| **Who** | 谁受益？谁使用？ | target_users | 法务部3名专员+运营部5名员工 |
| **What** | 要做什么？解决什么？ | demand_name, pain_points | 自动化采集侵权专利，替代手动 |
| **When** | 何时使用？何时截止？ | usage_scenario, deadline_risk | 日常使用，下月底前上线 |
| **Where** | 在什么场景？什么系统？ | usage_scenario | 现有专利库系统，内部OA |
| **Why** | 为什么要做？业务价值？ | background, expected_value | 已有2次侵权警告，手动太慢易漏 |
| **How** | 如何实现？资源投入？ | resource_input | 1人月开发，Python脚本 |

### 2. User Story 标准格式

将需求转化为标准用户故事：

```
作为 [目标用户角色]
我想要 [具体功能/改变]
以便 [实现的业务价值/用户价值]
```

示例：
```
作为 法务部专员
我想要 自动化采集亚马逊侵权专利数据
以便 及时发现侵权风险，避免业务损失
```

### 3. JTBD（Jobs-to-be-Done）深挖

区分表面需求和深层Job：

| 表面需求 | 深层Job |
|----------|---------|
| "做个自动化采集脚本" | "避免侵权风险导致的业务损失" |
| "改个按钮位置" | "提升日常操作效率" |
| "加个报表功能" | "辅助管理层快速了解业务状况" |

补全时，AI需要识别：
- 用户的**核心Job**是什么？
- 这个Job的**情感性**需求是什么？（安全感、效率感）
- 这个Job的**社会性**需求是什么？（被认可、专业形象）

### 4. PR-FAQ 反推验证

从客户视角反推需求真实性：

**Press Release 标题**: 这个需求如果成功，新闻稿标题是什么？
- "法务部侵权预警系统上线，侵权风险降低80%"
- "运营效率提升3倍，采集脚本自动化"

**FAQ 关键问题**:
- 有多少客户需要？ → 量化用户规模
- 他们当前如何解决？ → 识别当前方案不足
- 我们能做到吗？ → 评估技术可行性

### 5. SMART 价值量化

确保预期价值符合 SMART 原则：

| 原则 | 要求 | 补全前 | 补全后 |
|------|------|--------|--------|
| Specific | 具体 | "提升效率" | "采集效率从50件/天提升至150件/天" |
| Measurable | 可量化 | "减少风险" | "侵权警告从每季度2次降至0次" |
| Achievable | 可实现 | "完全自动化" | "覆盖80%常见场景，剩余人工复核" |
| Relevant | 相关 | "做个脚本" | "直接关联法务合规目标" |
| Time-bound | 有时限 | "尽快上线" | "2026年6月30日前上线" |

## 三、补全输出模板

补全完成后，需求描述应包含以下完整信息：

```markdown
## 需求：{demand_name}

### 基本信息
- 需求编号: {demand_id}
- 需求分类: {demand_category}
- 提交人: {submitter}

### 背景与痛点
{background}
{pain_points}

### 目标用户
{target_users}

### 使用场景
{usage_scenario}

### 用户故事
As a {角色}, I want {功能}, so that {价值}

### 预期价值
{expected_value}

### 跨部门影响
{cross_department_impact}

### 资源投入
{resource_input}

### 紧急程度
{deadline_risk}

### 成功指标
{success_metrics}

### 风险与假设
{risks_assumptions}

### 信息置信度
- 明确提供: [用户明确说的]
- 合理推测: [AI推断的]
- 需要确认: [AI假设但需用户确认的]
```

---

## 四、补全质量自检

补全完成后，AI 需要自检：

1. **必须字段**（缺则评分不可靠）：
   - [ ] background: 描述了当前状况和问题
   - [ ] pain_points: 至少1个具体痛点
   - [ ] target_users: 明确目标用户
   - [ ] expected_value: 包含可量化价值
   - [ ] usage_scenario: 描述了使用场景

2. **建议字段**（有则更专业）：
   - [ ] success_metrics: 定义了成功标准
   - [ ] risks_assumptions: 列出了风险
   - [ ] resource_input: 估算了资源投入

3. **信息标记**：
   - [ ] 标记了哪些是用户明确提供的
   - [ ] 标记了哪些是AI推测的
   - [ ] 标记了哪些需要用户确认

## 五、补全示例

### 输入（用户原始描述）
```
帮我评估这个需求：做个亚马逊侵权专利库的自动化采集脚本
```

### 输出（AI补全后）
```
## 需求：亚马逊侵权专利库自动化采集系统

### 基本信息
- 需求编号: D001
- 需求分类: 合规风控 + 效率提升
- 提交人: [待确认]

### 背景与痛点
当前依赖手动方式采集亚马逊侵权专利数据，存在以下问题：
1. 采集效率低：人工采集速度慢，无法覆盖全部专利
2. 容易遗漏：人工比对容易遗漏相似专利
3. 已有2次侵权警告：表明风险正在发生

### 目标用户
- 法务部：负责侵权风险评估和处理
- 运营部：负责产品上架前的专利排查
- 预计影响用户：法务3人 + 运营5人 = 8人

### 使用场景
- 日常：运营人员在新品上架前，使用脚本自动检索相关专利
- 定期：法务人员定期运行脚本，生成侵权风险报告
- 触发：发现疑似违规时，快速检索确认

### 用户故事
As a 运营专员
I want 自动检索亚马逊侵权专利库并生成风险报告
So that 在商品上架前发现并规避侵权风险

### 预期价值（SMART量化）
- Specific: 替代手动采集，实现自动化检索和比对
- Measurable: 采集效率从50件/天提升至150件/天（提升3倍），侵权警告从每季度2次降至0次
- Achievable: 覆盖80%常见专利类型，剩余复杂案例人工复核
- Relevant: 直接关联公司合规风控目标
- Time-bound: 2026年6月30日前上线

### 跨部门影响
- 法务部：减少手动比对工作量，提高风险评估效率
- 运营部：新品上架前自动专利排查，减少侵权风险
- 可能涉及技术部：提供数据采集基础设施支持

### 资源投入
- 预计1人月开发
- 技术栈：Python爬虫 + 专利数据库API
- 外部依赖：亚马逊专利数据接口可用性

### 紧急程度
- 已有2次侵权警告，存在合规风险
- 建议下月底前上线
- 延期后果：可能产生更多侵权纠纷和业务损失

### 成功指标
- 采集效率提升≥3倍
- 侵权警告数降至0
- 用户满意度≥4/5
- 覆盖≥80%常见专利类型

### 风险与假设
- 技术风险：亚马逊API可能变更，需定期维护
- 业务风险：专利比对准确率可能不足，需人工复核
- 假设：专利数据接口稳定可用

### 信息置信度
- 明确提供: 自动化采集、提升效率、避免侵权
- 合理推测: 法务+运营使用、1人月投入、下月底截止
- 需要确认: 具体用户人数、技术栈选择、外部依赖
```
