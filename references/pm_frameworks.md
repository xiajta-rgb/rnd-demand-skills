# 产品管理专业框架参考文档

本文档详细说明融入评分机制的8大产品管理框架，指导AI在评分时如何运用这些专业视角进行深度分析。

---

## 一、RICE 优先级框架

**来源**: Intercom 产品团队 (2018)
**核心思想**: 通过 Reach、Impact、Confidence、Effort 四个维度量化需求优先级

### 框架要素

| 要素 | 含义 | 评分映射 |
|------|------|----------|
| **Reach** | 在时间周期内影响多少用户/客户 | → 影响范围 |
| **Impact** | 对每个受影响用户/客户的影响程度 (3=大量, 2=中等, 1=轻微, 0.5=极小) | → 业务价值 |
| **Confidence** | 对 Reach/Impact 估计的信心度 (100%=高, 80%=中, 50%=低) | → 投入产出比 |
| **Effort** | 需要多少人月 (person-months) | → 实施复杂度 |

### RICE 评分公式

```
RICE Score = (Reach × Impact × Confidence) / Effort
```

### 在需求评审中的应用

**高RICE需求特征**:
- 覆盖大量用户 (Reach高)
- 对每个用户产生显著影响 (Impact高)
- 有数据支撑，信心充足 (Confidence高)
- 实现成本低 (Effort低)

**低RICE需求特征**:
- 仅影响少数人 (Reach低)
- 影响微弱 (Impact低)
- 基于假设，缺乏验证 (Confidence低)
- 实现成本高 (Effort高)

### 评分指导

- 当需求描述中包含具体用户数量、影响比例时 → Confidence高，评分偏上限
- 当需求描述仅含模糊表述如"可能有帮助" → Confidence低，评分偏下限
- 当需求明确提到"1人月可完成" → Effort低，实施复杂度评分偏上限
- 当需求提到"需要多团队配合" → Effort高，实施复杂度评分偏下限

## 二、Jobs-to-be-Done (JTBD)

**来源**: Clayton Christensen (哈佛商学院), Tony Ulwick (Strategyn)
**核心思想**: 用户不是购买产品，而是"雇佣"产品来完成某个任务(Job)

### 框架要素

| Job类型 | 含义 | 示例 |
|---------|------|------|
| **功能性Job** | 用户需要完成的核心任务 | "快速找到侵权专利" |
| **情感性Job** | 用户想要的感觉 | "不再担心侵权风险" |
| **社会性Job** | 用户希望被他人如何看待 | "被团队认可为高效的专业人士" |

### JTBD 分析方法

1. **识别核心Job**: 这个需求要帮用户完成什么任务？
2. **评估Job重要性**: 这个Job对用户有多重要？(核心 vs 边缘)
3. **评估当前满意度**: 用户当前如何完成这个Job？满意度如何？
4. **机会得分**: 重要性高 + 当前满意度低 = 高机会

```
机会得分 = 重要性 - 当前满意度
```

### 在需求评审中的应用

**核心Job需求** (高分):
- 需求直接解决用户的核心任务
- 当前解决方案满意度极低
- 机会得分 > 6 (10分制)

**边缘Job需求** (低分):
- 需求仅解决用户的次要任务
- 当前已有较好解决方案
- 机会得分 < 3

### 评分指导

- 识别需求背后的核心Job: "做X" 背后真正的Job是什么？
- 如果核心Job是"避免风险/损失" → 业务价值偏上限
- 如果核心Job是"个人便利" → 业务价值偏下限
- 如果当前无替代方案 → 业务价值和紧急程度均偏上限
- 如果已有成熟替代方案 → 业务价值偏下限

## 三、Opportunity Solution Trees (机会解决方案树)

**来源**: Teresa Torres (Product Talk)
**核心思想**: 从期望结果出发，系统化地探索机会空间和解决方案

### 框架结构

```
期望结果 (Desired Outcome)
  ├── 机会1 (Opportunity)
  │     ├── 解决方案A
  │     ├── 解决方案B
  │     └── 解决方案C
  ├── 机会2 (Opportunity)
  │     ├── 解决方案D
  │     └── 解决方案E
  └── 机会3 (Opportunity)
        └── 解决方案F
```

### 在需求评审中的应用

**主干机会需求** (高分):
- 需求直接服务于当前最重要的期望结果
- 位于机会树的主干路径上
- 解决方案经过假设验证

**分支机会需求** (中等):
- 需求服务于次优先的期望结果
- 位于机会树的分支路径
- 有一定价值但非当前重点

**脱离机会树的需求** (低分):
- 需求与当前任何期望结果无关
- 属于"灵光一现"式需求
- 缺乏与业务目标的系统性关联

### 评分指导

- 判断需求是否在当前产品/团队的机会树上
- 如果需求在主干上 → 战略匹配度评分偏上限
- 如果需求在分支上 → 战略匹配度评分中等
- 如果需求脱离机会树 → 战略匹配度评分偏下限
- 结合OKR判断：机会树的主干通常对齐公司级OKR

## 四、Amazon Working Backwards (逆向工作法)

**来源**: Amazon 内部产品开发方法论
**核心思想**: 从客户出发，先写新闻稿/FAQ，再决定是否开发

### 核心文档

| 文档 | 目的 | 关键问题 |
|------|------|----------|
| **Press Release** | 描述客户价值 | 客户为什么要用？解决了什么问题？ |
| **FAQ** | 回答关键疑问 | 有多少客户需要？我们能做到吗？ |
| **Customer Experience** | 描绘使用体验 | 客户如何发现、使用、获益？ |
| **User Stories** | 定义功能需求 | 作为X，我想要Y，以便Z |

### Working Backwards 六步法

1. 写新闻稿（面向客户的价值主张）
2. 写FAQ（回答内部和外部疑问）
3. 定义客户体验
4. 写用户故事
5. 评估技术可行性
6. 决定是否推进

### 在需求评审中的应用

**通过Working Backwards验证的需求** (高分):
- 能清晰写出客户价值主张
- 有明确的客户需求证据
- 客户体验路径清晰
- 技术可行性已评估

**未通过验证的需求** (低分):
- 无法清晰描述客户价值
- 基于内部假设而非客户需求
- 客户体验模糊
- 技术可行性未评估

### 评分指导

- 评估时思考：如果为这个需求写一篇新闻稿，标题是什么？
- 如果标题令人兴奋且客户导向 → 业务价值偏上限
- 如果标题只是"优化了X功能" → 业务价值偏下限
- 如果需求有客户访谈/调研支撑 → 类似Amazon的FAQ验证，Confidence高
- 如果需求来自内部想象 → Confidence低

## 五、OKR 对齐框架

**来源**: Intel (Andy Grove), 推广于 Google/LinkedIn
**核心思想**: 目标与关键结果对齐，确保每个需求都服务于更高层目标

### OKR 层级

| 层级 | 周期 | 示例 |
|------|------|------|
| **公司级OKR** | 年度/半年度 | O: 成为行业领先平台, KR: 市占率提升至30% |
| **部门级OKR** | 季度 | O: 提升平台效率, KR: 处理时间缩短50% |
| **团队级OKR** | 季度 | O: 优化采集流程, KR: 采集效率提升3倍 |

### 在需求评审中的应用

**直接支撑公司级OKR** (高分):
- 需求直接贡献于公司级Objective或Key Result
- 影响公司核心业务指标
- 通常为A级战略型需求

**支撑部门级OKR** (中等):
- 需求支持部门级目标
- 对公司级OKR有间接贡献
- 通常为B级经营优化型需求

**无OKR对齐** (低分):
- 需求与任何OKR无关
- 属于"做了也不错"类需求
- 通常为C级或D级需求

### 评分指导

- 判断需求支撑的OKR层级
- 公司级OKR → 战略匹配度17-20分
- 部门级OKR → 战略匹配度13-16分
- 团队级OKR → 战略匹配度9-12分
- 无OKR对齐 → 战略匹配度0-8分
- 注意：需求可能同时支撑多个层级的OKR，取最高层级

## 六、北极星指标框架 (North Star Metric)

**来源**: Sean Ellis (GrowthHackers), 推广于 Airbnb/Spotify
**核心思想**: 定义一个最能反映产品为客户交付核心价值的单一指标

### 北极星指标三要素

| 要素 | 含义 | 示例 |
|------|------|------|
| **North Star Metric** | 衡量核心价值的单一指标 | "每周完成采集任务数" |
| **Input Metrics** | 驱动NSM的可控指标 | "采集脚本可用率"、"用户激活率" |
| **North Star Playbook** | 团队围绕NSM的工作方式 | 所有需求评估都问"是否推动NSM" |

### 在需求评审中的应用

**直接影响北极星指标** (高分):
- 需求能直接提升NSM数值
- 例如：NSM="每周完成采集任务数"，需求="自动化采集脚本" → 直接提升
- 业务价值评分偏上限

**间接影响北极星指标** (中等):
- 需求通过改善Input Metric间接影响NSM
- 例如：需求="优化采集脚本稳定性" → 提升可用率 → 间接提升NSM
- 业务价值评分中等

**与北极星指标无关** (低分):
- 需求不直接影响NSM或Input Metrics
- 例如：需求="更换界面配色方案" → 与采集效率无关
- 业务价值评分偏下限

### 评分指导

- 首先识别公司/产品的北极星指标
- 评估需求与NSM的关联程度：直接 > 间接 > 无关
- 直接关联 → 业务价值21-25分
- 间接关联 → 业务价值11-20分
- 无关联 → 业务价值0-10分

## 七、跨越鸿沟理论 (Crossing the Chasm)

**来源**: Geoffrey Moore (1991)
**核心思想**: 技术产品从早期采纳者到主流市场的跨越是最关键的挑战

### 市场采纳曲线

```
创新者 → 早期采纳者 → 【鸿沟】 → 早期多数 → 晚期多数 → 滞后者
(2.5%)    (13.5%)                 (34%)       (34%)       (16%)
```

### 在需求评审中的应用

**跨越鸿沟阶段的需求** (高分):
- 需求帮助产品从早期采纳者跨越到主流市场
- 面向早期多数用户的核心需求
- 通常具有战略意义

**早期采纳者阶段的需求** (中等):
- 需求服务于创新者和早期采纳者
- 有价值但尚未验证主流市场需求
- 需要谨慎评估投入

**滞后期需求** (低分):
- 需求服务于晚期多数或滞后者
- 市场已充分验证，竞争激烈
- 差异化价值有限

### 评分指导

- 判断需求服务的用户群体处于采纳曲线的哪个阶段
- 跨越鸿沟阶段 → 影响范围评分偏上限，战略匹配度偏高
- 早期采纳者阶段 → 影响范围评分中等，需要更多验证
- 滞后期 → 影响范围可能广但差异化低，需综合评估
- 结合RICE Reach：跨越鸿沟阶段的需求Reach通常在快速增长

## 八、技术债务评估框架 (Technical Debt Assessment)

**来源**: Ward Cunningham (1992), Martin Fowler (技术债务象限)
**核心思想**: 每个技术决策都有长期维护成本，需求评审必须考虑技术债影响

### 技术债务象限

| | 有意(Prudent) | 无意(Reckless) |
|------|------|------|
| **谨慎(Deliberate)** | "我们必须快速交付，但知道后果" | "我不知道更好的做法" |
| **疏忽(Inadvertent)** | "现在这样够用，以后再重构" | "代码怎么变成这样的？" |

### 在需求评审中的应用

**减少技术债的需求** (高分):
- 需求明确减少现有技术债务
- 提升系统可维护性和可扩展性
- 实施复杂度评分偏上限（反向计分）

**不增不减的需求** (中等):
- 需求对技术债无显著影响
- 标准开发，无特殊技术风险
- 实施复杂度评分中等

**增加技术债的需求** (低分):
- 需求引入新的技术债务
- 快速修复但增加长期维护成本
- 实施复杂度评分偏下限（反向计分，即分数低）

### 评分指导

- 评估需求对技术架构的影响
- 减少技术债 → 实施复杂度8-10分 + 业务价值额外加分
- 不增不减 → 实施复杂度5-7分
- 增加技术债 → 实施复杂度0-4分 + 业务价值需额外审查
- 注意：短期增加技术债但长期收益巨大的需求，需在理由中说明权衡

## 框架综合应用指南

### 评分时的框架应用顺序

```
1. JTBD 分析 → 理解需求背后的真实任务
2. Working Backwards → 从客户视角验证需求价值
3. North Star → 评估需求与核心指标的关系
4. OKR 对齐 → 确认需求与战略目标的对齐度
5. OST 定位 → 判断需求在机会树上的位置
6. RICE 量化 → 用数据量化优先级
7. Crossing the Chasm → 评估市场阶段影响
8. Technical Debt → 评估技术影响
```

### 框架交叉验证

当多个框架指向相同结论时，评分置信度高：
- JTBD核心Job + North Star直接关联 + OKR公司级对齐 → 确认A级战略型
- RICE高Score + Working Backwards通过验证 → 确认高优先级
- JTBD边缘Job + OST脱离 + 无OKR对齐 → 确认D级无效

当框架结论矛盾时，需要深入分析：
- JTBD重要但RICE Score低 → 可能是重要但资源不足，考虑分期
- OKR对齐但Working Backwards未通过 → 可能是战略正确但执行方案有误

### 输出中的框架证据

评分报告中应体现框架分析结果：

```
🔍 PM框架分析：
- JTBD: 核心Job="避免侵权风险"，当前满意度极低 → 业务价值加分
- RICE: Reach=高(法务+运营), Impact=3(大量), Confidence=80%, Effort=1人月 → 高优先级
- North Star: 直接关联"采集任务完成数" → 业务价值加分
- OKR: 支撑部门级OKR"提升合规效率" → 战略匹配度中等偏上
- Working Backwards: 客户强需求，已有2次侵权警告 → 验证通过
- OST: 位于"合规风控"机会树主干 → 战略匹配度加分
- Crossing the Chasm: 跨越鸿沟阶段需求 → 影响范围加分
- Tech Debt: 不增加技术债 → 实施复杂度中等
```
